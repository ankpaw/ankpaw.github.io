## 2026-04-08 - Added Keyboard Accessibility to Experience Timeline Cards
**Learning:** Custom interactive elements designed as cards (like the experience timeline accordions) often lack keyboard support (`tabIndex`, `role`, and `onKeyDown`) out of the box, rendering them unusable for keyboard and screen reader users. Simply adding an `onClick` handler to a `<div>` or custom UI card is insufficient for a11y.
**Action:** Always ensure that custom interactive elements acting as buttons have `role="button"`, `tabIndex={0}`, keyboard event handlers (for `Enter` and `Space`), and proper `aria-expanded`/`aria-controls` attributes if they toggle content. Ensure focus states are clear using `focus-visible`. Update click handler types (e.g. from `React.MouseEvent` to `React.SyntheticEvent`) to safely handle both mouse and keyboard inputs.
## 2026-04-11 - Improved Screen Reader Experience for State-Driven Icon Buttons
**Learning:** Adding an `aria-label` to a button replaces its inner text for screen readers. If a button (like a "Like" button) contains both an icon and a dynamic value (e.g., the number of likes), adding an `aria-label` solely for the icon will hide the dynamic value. Furthermore, relying only on the visible icon and number leaves the purpose of the button ambiguous to screen reader users (e.g., it reads "5, button" instead of "Like post, 5 likes"). Additionally, `disabled` states need a `title` or tooltip so mouse users understand *why* the action is blocked.
**Action:** When an interactive element like a toggle button contains visible state (e.g., counters), the `aria-label` must combine the action context *and* the value (`aria-label="Like post. 5 likes"`). Always include `aria-pressed` for toggle states, apply `aria-hidden="true"` to decorative/redundant icons, and add a `title` attribute to explain disabled states to sighted users.
