## 2026-04-08 - Added Keyboard Accessibility to Experience Timeline Cards
**Learning:** Custom interactive elements designed as cards (like the experience timeline accordions) often lack keyboard support (`tabIndex`, `role`, and `onKeyDown`) out of the box, rendering them unusable for keyboard and screen reader users. Simply adding an `onClick` handler to a `<div>` or custom UI card is insufficient for a11y.
**Action:** Always ensure that custom interactive elements acting as buttons have `role="button"`, `tabIndex={0}`, keyboard event handlers (for `Enter` and `Space`), and proper `aria-expanded`/`aria-controls` attributes if they toggle content. Ensure focus states are clear using `focus-visible`. Update click handler types (e.g. from `React.MouseEvent` to `React.SyntheticEvent`) to safely handle both mouse and keyboard inputs.
## 2026-04-10 - Enhanced A11y for State-Driven Buttons
**Learning:** Icon-only and state-driven buttons (like a "Like" toggle or conditionally disabled form fields) lack sufficient context for screen reader users and those navigating via keyboard when disabled. For example, screen readers might announce a disabled heart button merely as "button, unclickable" without clarifying *why* it's disabled. Additionally, visually toggled states (filled vs outlined icons) aren't communicated to assistive technologies automatically.
**Action:** Always implement `aria-pressed` for toggle buttons, use a comprehensive `aria-label` that completely overrides inner text to convey both the action and current state (e.g., `Unlike post, 5 likes`), hide decorative interior icons with `aria-hidden="true"`, and provide a `title` attribute on disabled elements to explain *why* the action is blocked (e.g., "Sign in to like this post").
